# 理解測試協議(Phase C 發布前閘門)

目的:在投入英文版與社群發布之前,驗證頁面真的能讓目標讀者「親手獲得 aha」。這是取代第三方分析工具的核心量測。

---

## 受測者(n=5)

- 3–4 位:零 ML 先備的好奇探索者(符合品牌受眾;系上非資電科系同學、朋友皆可)。
- 1–2 位:有部署/ML 背景的工程師(驗證進階摺疊層)。
- 排除:參與過本頁製作或已被劇透 aha 的人。
- 招募管道與名單:凜空填。

## 流程(每人 15–20 分鐘,線上或現場)

1. **自由探索(5–8 分鐘)**:只給連結,不給任何說明。請受測者放聲思考(think-aloud),觀察者只記錄不引導。手機受測至少 2 位。
2. **自述題**:「用你自己的話,說說這一頁想讓你懂什麼?」逐字記錄。
3. **預測題(aha 的可遷移性)**:問一個頁面**沒有直接展示**的情境,例:「如果把『保護異常值』的比例從 1% 降到 0.1%,你猜會發生什麼?為什麼?」——答對方向且理由正確才算過。
4. **卡點回放**:請受測者指出最困惑的一處(記入返工清單)。
5. 工程師受測者加問:進階摺疊區有無錯誤或想吐槽之處(逐字記錄)。

## 通過標準

- **≥ 4/5 通過**,單人通過定義 = 自述題命中核心 aha(允許用語不精確)**且**預測題方向正確。
- 任何受測者指出的**技術錯誤**:無論人數,一律修正後才可發布(回 claim-source 表補記)。
- 未達 4/5:針對卡點返工,更換受測者重測;連續兩輪未過 → 上報為旗艦設計問題,重開 go/no-go。

## 誠實條款

- 觀察者(凜空)全程不解釋、不救場;受測者卡住就是資料。
- 「停留很久」不算好訊號——可能是看不懂;以自述與預測為準。
- 結果原始記錄存 `/retros/`,與製作履歷互相引用。
