### A — 資料匯：Analytics Engine vs D1 表（**架構決定性，且牽動預算，必先定**）
ROADMAP 寫「Analytics Engine 綁定 `WONDER_EVENTS`」。但：
- **Analytics Engine 可能需 Workers Paid（US$5/月）**——與創辦人錨點「**無預算**」衝突（需你在 dashboard 確認 AE
  是否在你目前免費方案可寫入；earlier 截圖顯示 Requests 0/100,000＋Upgrade 鈕，疑似 Free）。
- **提案①（D1 events 表，推薦）**：直接沿用**已建好、免費**的 D1 `neblux-echo`，加一張 `event` 表
  （`tour TEXT, event TEXT, lang TEXT, step INTEGER, returning INTEGER, ts INTEGER`）。查詢走 D1 Console／
  `wrangler d1 execute` 的 SQL——**零新成本、零新 `[人工]` 綁定**（DB 已綁）。代價：D1 有寫入配額（免費 100k
  writes/日），本站流量遠低於此、無虞；高流量才需 AE 的取樣擴展性。**吻合「次要資產＋無預算」。**
- **提案②（Analytics Engine，照 ROADMAP 字面）**：`env.WONDER_EVENTS.writeDataPoint(...)`，內建取樣、適合高量。
  但要 `[人工]` 建 AE 綁定＋**可能付費**＋查詢要另用 AE SQL API（帳號 token，離線）＋完走率 SQL 需 `SUM(_sample_interval)`
  還原取樣。對現階段流量是殺雞用牛刀。
- **我的建議：①（D1 表）**。省錢、省一個 `[人工]`、複用現有 D1、查詢就在同一個 Console。ROADMAP 的 AE 視為
  「日後流量夠大再遷移」的選項，不是現在的鐵板。**若你確認 AE 免費可用且想要它的擴展性，才走 ②。**

### B — hook 轉換率的粒度（picker 曝光事件）
完走率＝finish/start 很直接。但「hook 轉換率」要「picker 曝光→start」，需在 picker 補曝光事件。
- **提案①（單一 `picker_view`，推薦、貼 ROADMAP「補一個 view 事件」）**：`renderPicker()` 觸發一次
  `event=picker_view`（不帶 tour）。轉換率＝(任一 tour start 數)/(picker_view 數)＝**整體**「看到清單→開始走」比率。便宜。
- **提案②（per-tour impression）**：picker 每列曝光各發一事件（帶 tour），可歸因到**具體哪張 hook 卡**轉換好。
  精準但事件量 ×19/次載入、且多數趟低樣本雜訊大。
- **我的建議：①**。先要整體漏斗；per-tour hook A/B 是有流量後的細活（少而精）。

### C — 回墜率的匿名實作（**隱私關鍵**）
「回墜率」＝同 session 第二次 start 的比率（讀者走一趟後又回來開另一趟＝黏著）。難點：**不得存 session 識別**。
- **提案（推薦）**：純 client 端 `sessionStorage['wonder-started']` 布林旗標。start 時：若旗標已存在→本次 start
  事件帶 `returning=1`，否則 `returning=0`＋設旗標。**伺服器端不存任何 session id，只收到一個布林**——回墜率＝
  `returning=1` 的 start 佔比。完全匿名、無跨 session 追蹤（關瀏覽器即清）。
- 替代：不做回墜率（減一欄）。→ 少一個指標，但這是 DIRECTION 想要的黏著訊號，建議保留。

## 審查後設計修訂（2026-07-08，凜空裁 Finding #1）
原 brief 設計為 append-only `event` 表（含 ms `ts`）。對抗審查指出：低流量下逐事件列＋ms ts＋插入
順序可重建個人 session（違鐵律 2）。**凜空裁決改為純聚合計數表**（同 echo 的 counter 模式）：
- `event_count(event, tour, lang, step, is_return, count)`，PK＝前五者，UPSERT +1。**無 ts、無逐事件
  列、無順序**——徹底無法重建個人。sentinel：picker_view `tour=''`、start/picker_view `step=0`。
- 代價：無時間維度（時間趨勢改離線快照整表，Phase 3 素材）。查詢範本改 `SUM(CASE…count)`。
- 前端埋點不變（仍發同樣 payload）；只有後端存法從 INSERT 逐列改 UPSERT 計數。
